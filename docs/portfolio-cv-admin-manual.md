# Portfolio CV and Admin Manual

This manual explains where each part of the public CV/portfolio page comes from, what content feeds the Executive Summary PDF and AI answers, and what each admin section controls.

## Main Data Sources

The portfolio uses several data stores and code-level content sources:

- `profile` table in the per-user portfolio database, accessed through `db.portfolio(user)`.
- `cv_context` table, a flexible key/value content table used for public page copy and AI context.
- `experiences` table, used for the public timeline, PDF, AI chat, and JD analyser when `is_cv_context = 1`.
- `skills` table, used for visible skill cards and AI/JD context.
- `gaps`, `faqs`, and `ai_instructions` tables, used by the portfolio AI chat.
- `jd_submissions` table, used to store Candidate Analysis submissions and generated answers.
- `skill_candidates` table, auto-detected from CV-tagged Hub projects and optionally promoted into skills or portfolio-chat-only context.
- Hub `projects`, `messages`, and `documents`, where projects marked as CV context feed the public portfolio chat.
- `lib/aiBuilds.js`, which currently provides the AI-Assisted App Builds section and content.

## Executive Summary PDF

The Executive Summary PDF is generated by `GET /executive-summary` in `routes/portfolio.js`.

It pulls from:

- `profile.full_name`, with fallback to the site user name.
- `cv_context.role_label` or `profile.current_title`.
- `cv_context.summary` or `profile.elevator_pitch`.
- Portfolio URL, generated from the current user: `douglas.mclellan.scot` or `nakai.mclellan.scot`.
- `profile.email`, with Douglas fallback email.
- `profile.phone_public`, with Douglas fallback phone.
- `profile.location`.
- `cv_context.linkedin_url`.
- First five `experiences` where `is_cv_context = 1`, ordered by `display_order`.
- For Douglas only, hardcoded/shared project content from `lib/aiBuilds.js`.

Important notes:

- `profile.remote_preference`, `availability_status`, `available_from`/notice period, and salary fields are not currently printed in the PDF.
- The helper `getExecutiveSummaryText()` also includes availability and remote preference, but the current route generates the PDF through `buildExecutiveSummaryPdf()`, not that text helper.
- The PDF does not include gaps, FAQs, values, culture preferences, Hub project memories, or AI instructions.

## Portfolio AI Chat

The public chat drawer and `/chat` page call `POST /api/chat`.

The AI chat context includes:

- `profile` fields: name, email, title, location, target titles, company stages, elevator pitch, career narrative, what you are looking for, what you are not looking for, management style, work style, salary range/currency, availability, notice period, and remote preference.
- All `cv_context` rows.
- Douglas AI-Assisted App Builds from `lib/aiBuilds.js` if the `cv_context` row is missing.
- `experiences` where `is_cv_context = 1`.
- All `skills`, including level, self-rating, years, last used, evidence, and honest notes.
- All `gaps`.
- Values and culture fields from `profile`: must-haves, dealbreakers, management preferences, team size preferences, conflict handling, ambiguity handling, and failure handling.
- All `faqs`, ordered with common questions first.
- `skill_candidates` where `status = 'chat_only'`.
- Recent assistant messages from Hub projects where `projects.is_cv_context = 1`.
- `ai_instructions`.
- Recent conversation history for that chat session.

The system prompt tells the AI to speak in first person as the candidate, avoid fabrication, be honest about weak fit, and use the configured honesty level.

Important notes:

- This is the richest context path in the app.
- It sees backend-only preference data even where the public page does not.
- It does not use web search.
- It stores visitor and assistant messages in `portfolio_messages`.

## Candidate Analysis / JD Analyser

The Candidate Analysis form on the public page calls `POST /api/analyse-jd`.

The JD analyser context includes:

- All `cv_context` rows.
- Douglas AI-Assisted App Builds from `lib/aiBuilds.js` if the `cv_context` row is missing.
- `experiences` where `is_cv_context = 1`.
- `skills` using only name, level, and category.
- The pasted job description.
- Contact block added after the answer, using `profile.email` and `profile.phone_public`.

It stores:

- The submitted job description.
- The generated AI response.
- Created timestamp.

Important notes:

- It does not currently include `profile` fields such as remote preference, salary, looking-for/not-looking-for, career narrative, availability, or management/work style.
- It does not include gaps, FAQs, values/culture, AI instructions, Hub project memories, or chat-only skill candidates.
- Skill candidates only affect the JD analyser after they are promoted into the `skills` table with “Add to JD Analyzer”.

## Public CV Page Sections

### Navigation

Displayed links:

- Portfolio.
- Experience.
- AI Builds, shown only when `aiBuildProjects` exists, currently Douglas only.
- Capability.
- Contact.

Source:

- Mostly static template.
- AI Builds link depends on `lib/aiBuilds.js` data being passed by the route.

### Hero

Fields:

- Name: derived from active site user, not profile.
- Credentials: `cv_context.credentials`; Nakai has fallback credentials.
- Main heading: `cv_context.hero_title`.
- Tagline: `cv_context.tagline`.
- Portrait: `cv_context.portrait_url`.
- Availability details under the portrait:
  - `profile.availability_status`, shown only when an option is chosen.
  - `profile.available_from`, now used as the notice period and shown only when selected.
  - `profile.remote_preference`, shown only when selected.
- Executive Summary PDF button: static link to `/executive-summary`.
- Contact button: static anchor to footer.

Fallbacks:

- Douglas fallback hero title: “Product & Technology Leader.”
- Douglas fallback tagline is generic AI/developer-tooling language and may not match the current IT/security positioning if the DB row is absent.

### Candidate Analysis

Visible content:

- Static explanatory copy in `views/portfolio/index.ejs`.
- Textarea for a role description.
- “Assess candidate fit” button.
- Result area rendered from streamed Markdown.

Source:

- Static page text.
- Runtime API answer from `/api/analyse-jd`.

Stored:

- Each submission and answer is stored in `jd_submissions`.

### Vision Band

Fields:

- Title: `cv_context.vision_title`.
- Body paragraphs: `cv_context.vision_body1` and `cv_context.vision_body2`.
- Ask About Me button: opens chat drawer or `/chat` on mobile.

Fallbacks:

- Douglas fallbacks are generic product/AI language if DB rows are missing.

### AI-Assisted App Builds

Visible only for Douglas.

Source:

- `lib/aiBuilds.js`.

Visible fields:

- Section intro.
- Project cards for Bowling Load Tracker, McLellan Hub, and MRVR.
- GitHub links.
- Summary text.
- Stack text.
- First three bullets per project.

Also surfaced in:

- `cv_context.ai_assisted_app_builds` via migration if missing.
- Portfolio AI chat context.
- JD analyser context.
- Executive Summary PDF, as a shorter section.
- Seed script.

### Experience Matrix

Source:

- `experiences` where `is_cv_context = 1`, ordered by `display_order`.

Visible fields:

- `start_date`.
- `end_date` or “Present”.
- `role`.
- `company`.
- `description`.

Also surfaced in:

- Executive Summary PDF, first five rows.
- Portfolio AI chat.
- JD analyser.

### Capability & Innovation

Visible content:

- Strong Proficiency card uses the first three `skills` where `level = 'strong'`.
- AI & Analytics card is static.
- Toolkit / Candidate Analysis card is static.
- Strategic Focus bar is static text.

Source:

- Only the first card is database-driven.
- Other cards currently do not read from admin fields.

Also surfaced in:

- Skills feed portfolio AI chat and JD analyser more broadly than the visible first-three display.

### Governance Essay

Fields:

- Heading: `cv_context.essay_title`.
- Body: `cv_context.essay_body`.
- Stats: `cv_context.stat1_num`, `stat1_label`, `stat2_num`, `stat2_label`.

Static content:

- The two image/tile blocks are hardcoded: “Security-first delivery” and “Useful leadership signal”.

### Footer / Contact

Fields:

- Footer identity: active site user name and `cv_context.role_label`.
- Email: `profile.email`, with Douglas fallback email.
- Phone: `profile.phone_public`, encoded in page markup and revealed client-side.
- LinkedIn: `cv_context.linkedin_url`.
- Executive Summary PDF link.

Important notes:

- Location is not visible in the footer.
- Availability status, notice period, remote preference, and salary are not visible in the footer.

## Admin Area

Admin access is through `/admin`, authenticated through Google OAuth. The admin page is a tabbed interface rendered by `views/admin/index.ejs`.

### Profile Tab

Purpose:

- Main candidate profile and job-search metadata.

Fields and surfacing:

- `full_name`: AI chat profile context; Executive Summary PDF title fallback. Public page still derives name from the active site user.
- `email`: footer email, AI chat context, JD analyser contact block, Executive Summary PDF.
- `current_title`: AI chat context; Executive Summary PDF fallback if `cv_context.role_label` is absent.
- `location`: AI chat context and Executive Summary PDF.
- `phone_public`: footer reveal button, JD analyser contact block, Executive Summary PDF.
- `target_titles`: AI chat context only.
- `target_company_stages`: AI chat context only.
- `elevator_pitch`: AI chat context; Executive Summary PDF fallback if `cv_context.summary` is absent.
- `career_narrative`: AI chat context only.
- `looking_for`: AI chat context only.
- `not_looking_for`: AI chat context only.
- `management_style`: AI chat context only.
- `work_style`: AI chat context only.
- `salary_min`, `salary_max`, `salary_currency`: AI chat context only.
- `availability_status`: shown under the portrait when selected; AI chat context; text helper for Executive Summary, but not current PDF.
- `available_from`: used as notice period; shown under the portrait when selected; AI chat context.
- `remote_preference`: shown under the portrait when selected; AI chat context; text helper for Executive Summary, but not current PDF.

Backend-only or partially surfaced:

- Hybrid/remote/on-site preference is shown under the portrait when selected.
- Salary is not shown publicly, but is available to the public AI chat.
- Target titles/company stages are not shown publicly, but are available to the public AI chat.
- Looking-for/not-looking-for are not shown publicly, but are available to the public AI chat.

### Experience Tab

Purpose:

- Manage timeline roles and AI/PDF/JD experience evidence.

Fields:

- `company`: visible in public timeline, PDF, AI chat, JD analyser.
- `role`: visible in public timeline, PDF, AI chat, JD analyser.
- `start_date`: visible in public timeline, PDF, AI chat, JD analyser.
- `end_date`: visible in public timeline, PDF, AI chat, JD analyser; blank becomes “Present”.
- `description`: visible in public timeline, PDF, AI chat, JD analyser.
- `is_cv_context`: controls whether the role appears on the public timeline and feeds PDF, AI chat, and JD analyser.
- `display_order`: controls ordering.

Gap:

- Existing experience rows can be deleted, but this admin screen does not currently expose inline editing for existing experience content.

### Skills Tab

Purpose:

- Manage skills used visibly and by AI.

Fields:

- `name`: visible in public strong-skills card if level is strong and in the first three; AI chat; JD analyser.
- `level`: visible in admin; controls public strong-skills card when `strong`; AI chat; JD analyser.
- `category`: AI chat and JD analyser context, but not visible publicly.
- `display_order`: controls ordering.
- `self_rating`: AI chat context only.
- `evidence`: AI chat context only.
- `honest_notes`: AI chat context only.
- `years_experience`: AI chat context only.
- `last_used`: AI chat context only.

Important notes:

- The public page shows only the first three strong skills.
- JD analyser only receives name, level, and category. It does not receive self-rating, evidence, honest notes, years, or last-used.

### Detected Topics Tab

Purpose:

- Scan CV-tagged Hub projects and detect repeated technical/skill topics.

Source:

- Hub `messages` and `documents` attached to projects where `projects.is_cv_context = 1`.
- Detection patterns in `lib/cvCandidates.js`.

Actions:

- Rescan CV context: reruns detection.
- Add to JD Analyzer: creates or updates a `skills` row and marks the candidate `promoted_jd`.
- Add to portfolio chat only: marks candidate `chat_only`, which feeds only the portfolio AI chat.
- Dismiss: marks candidate `dismissed`.
- Reset to pending: makes it actionable again.

Surfacing:

- `promoted_jd`: becomes a skill, so it can feed public strong-skills display, portfolio AI chat, and JD analyser.
- `chat_only`: feeds portfolio AI chat as “Emerging topics from CV context”.
- `dismissed`: not surfaced.

### Gaps Tab

Purpose:

- Store honest limitations, mismatches, or development areas.

Fields:

- `gap_type`: AI chat context.
- `description`: AI chat context.
- `why`: AI chat context.
- `interested_in_learning`: AI chat context.
- `display_order`: stored and used for ordering, but not exposed in the add/edit UI except as hidden for existing rows.

Surfacing:

- Portfolio AI chat only.

Not surfaced:

- Public page.
- Executive Summary PDF.
- JD analyser.

### Values & Culture Tab

Purpose:

- Store preference and working-style context.

Fields:

- `must_haves`: portfolio AI chat only.
- `dealbreakers`: portfolio AI chat only.
- `mgmt_prefs`: portfolio AI chat only.
- `team_size_prefs`: portfolio AI chat only.
- `conflict_handling`: portfolio AI chat only.
- `ambiguity_handling`: portfolio AI chat only.
- `failure_handling`: portfolio AI chat only.

Not surfaced:

- Public page.
- Executive Summary PDF.
- JD analyser.

### FAQ Tab

Purpose:

- Give the public portfolio chat consistent answers to common questions.

Fields:

- `question`: portfolio AI chat.
- `answer`: portfolio AI chat.
- `is_common`: orders common questions first in AI context.
- `display_order`: used for ordering.

Not surfaced:

- Public page.
- Executive Summary PDF.
- JD analyser.

### AI Instructions Tab

Purpose:

- Control how the public portfolio chat behaves.

Fields:

- `honesty_level`: changes the chat tone descriptor from diplomatic/polished through balanced, direct, or brutally honest.
- `ai_instructions.text`: added to the portfolio AI chat system prompt.
- `display_order`: accepted by the route, but the current add form does not expose it.

Surfacing:

- Portfolio AI chat only.

Not surfaced:

- Public page.
- Executive Summary PDF.
- JD analyser.

### Candidate Analysis Submissions Tab

Purpose:

- Review submitted role descriptions and generated recruiter-facing assessments.

Fields shown:

- `created_at`.
- `job_description`.
- `ai_response`.

Source:

- Automatically populated by public Candidate Analysis form.

Actions:

- Read-only in current admin UI.

### CV Copy Tab

Purpose:

- Edit flexible page copy and broad AI context rows.

Important section keys used visibly on the public page:

- `role_label`: footer and Executive Summary title fallback.
- `hero_title`: hero heading.
- `credentials`: hero credential chip.
- `tagline`: hero supporting copy.
- `portrait_url`: hero image.
- `vision_title`: vision band heading.
- `vision_body1`: first vision paragraph.
- `vision_body2`: second vision paragraph.
- `essay_title`: essay heading.
- `essay_body`: essay paragraph.
- `stat1_num`, `stat1_label`, `stat2_num`, `stat2_label`: essay stats.
- `linkedin_url`: footer link and Executive Summary PDF.

Important section keys used by AI/PDF:

- `summary`: Executive Summary PDF.
- `ai_assisted_app_builds`: AI chat/JD analyser context if present; also inserted by migration for Douglas.
- Any other custom key: included in portfolio AI chat and JD analyser as raw CV copy.

Important notes:

- The public page only knows about specific keys. Adding a new key does not automatically make it visible on the page.
- The AI chat and JD analyser receive all keys, so new CV copy can influence AI even if it is not visually displayed.

### Hub Projects Tab

Purpose:

- Control which private Hub projects feed into public portfolio AI context and detected topic scanning.

Fields/actions:

- Project name and slug are shown from Hub `projects`.
- CV context flag toggles `projects.is_cv_context`.

Surfacing:

- Portfolio AI chat receives recent assistant messages from active CV-context projects.
- Detected Topics scanner reads messages and documents from active CV-context projects.

Not surfaced:

- Public page.
- Executive Summary PDF.
- JD analyser directly.

## Known Surfacing Gaps

These are not necessarily bugs, but they are places where backend/admin content exists without a visible public page surface.

- Remote/hybrid/on-site preference is stored, used by portfolio AI chat, and shown under the portrait when selected, but not included in the current Executive Summary PDF.
- Availability status and notice period are stored, used by portfolio AI chat, and shown under the portrait when selected, but not included in the current Executive Summary PDF.
- Salary range is stored and used by portfolio AI chat, but not shown publicly or included in the PDF.
- Target titles and target company stages are stored and used by portfolio AI chat, but not shown publicly or included in the JD analyser.
- Looking-for and not-looking-for are stored and used by portfolio AI chat, but not shown publicly or included in the JD analyser.
- Management style and work style are stored and used by portfolio AI chat, but not shown publicly or included in the JD analyser.
- Values/culture, gaps, FAQs, and AI instructions are rich AI-chat material, but they do not currently influence the JD analyser.
- Skills evidence, honest notes, years of experience, and last-used fields feed the portfolio chat, but the JD analyser receives only skill name, level, and category.
- Hub project CV context feeds public chat and detected topics, but not the public page, PDF, or JD analyser except indirectly through promoted skills.
- Public page “AI & Analytics”, “Toolkit”, “Strategic Focus”, and the two essay image tiles are static hardcoded content, not admin-driven.
- Existing experience rows can be deleted and new rows can be added, but existing experience rows are not currently editable through the admin UI.
- `ai_instructions.display_order` exists in storage and route handling, but the add-instruction form does not expose an order field.
- `gaps.display_order` exists and is used for ordering, but the add/edit UI does not meaningfully expose it.

## Quick Mental Model

- Public page: mostly `cv_context`, selected `experiences`, first three strong `skills`, profile contact details, and hardcoded project/static sections.
- Executive Summary PDF: profile basics, `cv_context.summary`, first five experiences, contact details, LinkedIn, and Douglas AI builds.
- Portfolio AI chat: broadest context; it sees profile preferences, all CV copy, experiences, detailed skills, gaps, values, FAQs, AI instructions, chat-only detected topics, and selected Hub memories.
- JD analyser: narrower recruiter screen; it sees CV copy, experiences, basic skills, AI builds, and the pasted job description.
- Admin: many fields are intentionally AI-context fields rather than visible-page fields.
